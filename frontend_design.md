# 前端设计规范（Frontend Design Spec）— 写作中心版

**版本**: v1.0  
**日期**: 2026-04-12  
**作者**: GitHub Copilot  
**状态**: Active - Canonical specification

> 本文是仓库前端的主规范文档，适用于后续重构与迭代。
>
> 设计目标：**保留当前浅色玻璃拟态风格，不改变视觉母语；仅将产品重心收束为“写作中心优先”的科研工作台。**
>
> 说明：`uiux` 技能仅作为设计判断框架，不作为项目运行时依赖，不需要接入代码层。

---

## 1. 概述

### 问题陈述

当前前端已经具备工作台骨架，但仍呈现出“多模块并列、写作区不绝对主导”的状态。主要问题是：

1. **写作区不是唯一焦点**：正文、导航、分析、技能、辅助面板的视觉权重接近。
2. **路由与导航未完全对齐**：部分入口存在展示但未闭环的问题。
3. **辅助能力过于显眼**：引用、建议、历史等信息容易打断写作心流。
4. **响应式约束不够明确**：窄屏与中屏下的主次关系还没有形成统一规则。
5. **状态设计不完整**：loading、empty、error、retry、offline、conflict 等状态需要统一定义。

### 设计目标

本规范希望把前端从“功能堆叠式工作台”升级为“写作中心优先”的科研工作台，同时保留现有浅色玻璃拟态风格。目标包括：

- 写作区成为视觉主角。
- 项目、资料、分析、技能保留为辅助层。
- 统一路由、布局、状态、动效、可访问性标准。
- 保持低噪、轻盈、克制的界面气质。

### 非目标

- 不切换为暗色主方案。
- 不引入纸张纹理、厚重米黄、复古文档风。
- 不把 `uiux` skill 做成前端运行时模块。
- 不删除项目总览、资料库、分析、技能等模块，只降低其视觉优先级。

## 2. 设计原则

### 2.1 视觉原则

- 保持现有浅色玻璃拟态（Light Glassmorphism）。
- 背景维持冷白、低噪、低饱和。
- 以“清透、克制、轻盈”为主，不引入重材质。
- 所有辅助内容都应弱化，不得压过正文写作区。
- 视觉语言要稳定、统一、可复用，不做一次性的炫技装饰。

### 2.2 产品原则

- **写作优先**：正文编辑区是第一层级。
- **导航清晰**：每个菜单项必须能对应真实路由或明确的 disabled 状态。
- **辅助克制**：AI 建议、引用、历史记录都应非阻塞、低干扰。
- **真实可用**：状态、错误、空态、保存、恢复都要有明确反馈。
- **功能分层**：主任务流、辅助任务流、低频任务流要有明确边界。

### 2.3 可访问性原则

- 普通文本对比度至少 `4.5:1`。
- 大号文本对比度至少 `3:1`。
- 所有交互必须具备 `focus visible`。
- 状态表达不能只依赖颜色。
- 必须支持 `prefers-reduced-motion`。
- 键盘用户必须能完成核心任务路径。

### 2.4 仓库规范原则

- 新增或更新的主规范文档应使用稳定、角色明确的文件名。
- 这类文档应作为长期可读的 canonical 文档保留在仓库根目录。
- 一次性产出的临时计划、报告、总结不应混入主规范。

## 3. 信息架构

### 3.1 页面层级

| 层级 | 页面/区域 | 角色 | 规则 |
|---|---|---|---|
| 一级 | 写作中心 | 主工作区 | 默认进入、默认可见、默认优先 |
| 一级 | 项目总览 | 入口页 | 负责项目切换与状态概览 |
| 二级 | 资料库 | 辅助资源 | 只在需要时展开 |
| 二级 | 分析页 | 结果与指标 | 不应占据主视觉 |
| 二级 | 技能页 | 能力入口 | 作为辅助能力展示，不是核心任务流 |

### 3.2 路由建议

- `/projects`：项目总览
- `/draft`：写作中心
- `/knowledge`：资料库
- `/analytics`：分析页
- `/skills`：技能页
- `/`：默认重定向到最近一次工作态，或项目总览
- `*`：统一转向合理入口，不要直接落空页

### 3.3 导航规则

- 导航项必须和真实页面对应。
- 未完成的模块要显示“未启用/开发中”，不要伪装成可用。
- 顶部、侧边栏、抽屉中的入口应保持一致。
- 菜单状态应和实际路由状态同步，不允许“高亮错位”。

## 4. 视觉规范

### 4.1 视觉基线

当前视觉应维持为：

- 冷白/偏灰的浅色背景
- 半透明玻璃表面
- 轻柔阴影
- 低噪边框
- 细腻留白
- 少量蓝紫色强调色

### 4.2 保留项

以下元素应视为当前视觉母语的一部分，原则上保留：

- 玻璃卡片和面板
- 低强度 backdrop blur
- 轻边框与柔阴影
- 统一的浅色主题
- 细腻的内容留白

### 4.3 不建议的方向

- 不要切换成纸张纹理主题。
- 不要引入厚重米黄背景。
- 不要把界面做成纯文档页。
- 不要把卡片做得太实、太重、太硬。
- 不要因为强调“写作感”而丢掉当前的工具型清晰度。

### 4.4 建议参数

| 项目 | 建议值 | 说明 |
|---|---|---|
| 写作区最大宽度 | 680–720px | 正文阅读与写作心流最优 |
| 顶部栏高度 | 56px 左右 | 轻量，不压迫正文 |
| 左侧导航展开宽度 | 240–280px | 桌面端稳定承载导航 |
| 右侧辅助栏宽度 | 320–360px | 适合建议/引用/历史 |
| 玻璃模糊 | 18–24px | 保持通透感 |
| 圆角 | 16–20px | 与现有风格一致 |
| 阴影 | 低强度软阴影 | 不制造压迫感 |

### 4.5 色彩语义

- 背景：保持浅色冷调。
- 主文本：深灰黑，接近印刷阅读感。
- 次级文本：中灰。
- 交互强调：蓝色系。
- AI / 引用提示：蓝紫色弱强调。

### 4.6 排版建议

- 正文优先使用系统 Sans-serif 或 `Inter` 作为编辑区字体。
- 行高建议维持在 `1.6–1.8`。
- 段落间距应明显但不过分松散。
- 不同层级标题要有稳定的字号与字重层次。

## 5. 交互规范

### 5.1 写作专注模式

当用户持续输入时：

- 左右侧栏轻微降显。
- 仅保留正文区的视觉权重。
- 不隐藏功能，只降低干扰。
- 停止输入后平滑恢复。

### 5.2 引用与来源

- 不要用大卡片占空间。
- 改成正文中的小型锚点、角标、微型标签。
- 点击后在右侧辅助区展开原文、来源、关联信息。
- 引用要“可追溯但不打断写作”。
- 在窄屏下，引用内容应自动降级为抽屉或底部面板。

### 5.3 AI 建议

- 建议内容必须是非阻塞式。
- 默认以轻提示出现，不抢光标。
- 仅在高置信度时显示更强提示。
- 支持一键接受、一键忽略、一键展开说明。
- 若建议与当前输入冲突，应以不打断写作为第一优先级。

### 5.4 状态设计

至少覆盖以下状态：

- loading
- empty
- error
- retry
- unsaved
- conflict
- read-only
- offline / degraded
- zen mode

### 5.5 动效规范

- 面板切换：180–220ms
- hover / focus：120ms
- 文本建议出现：120–160ms
- reduced motion：降为淡入淡出，不做滑动放大
- 动效不得影响输入响应或文本渲染稳定性

## 6. 技术规范

### 6.1 文件职责

| 文件 | 职责 | 技术要求 |
|---|---|---|
| `frontend/src/App.tsx` | 路由入口 | 负责重定向、路由分发、懒加载 |
| `frontend/src/layouts/MainLayout.tsx` | 全局骨架 | 负责 header、sidebar、响应式布局 |
| `frontend/src/components/DraftStudio.tsx` | 写作工作台容器 | 只做编排，不继续堆成巨型组件 |
| `frontend/src/index.css` | 主题与基础样式 | 集中维护颜色、玻璃、圆角、阴影、宽度 token |
| `frontend/src/contexts/WritingContext.tsx` | 写作状态 | 管理当前项目、章节、模式、面板状态 |
| `frontend/src/contexts/I18nContext.tsx` | 文案国际化 | 所有新增文案必须走字典 |
| `frontend/src/services/writingBackend.ts` | 资源接口 | 对接项目、章节、草稿、修订、资料、关联 |
| `frontend/src/services/runtimeClient.ts` | 运行时通信 | 处理事件流、会话、任务状态 |
| `frontend/src/types/resources.ts` | 后端资源类型 | 与服务端字段保持一致 |
| `frontend/src/types/writing.ts` | 前端视图模型 | 给工作台、面板、建议层使用 |
| `frontend/src/lib/simulationData.ts` | 模拟数据 | 仅做开发兜底，不混入生产主流程 |

### 6.2 建议拆分的组件

建议将 `DraftStudio` 拆成以下逻辑单元：

- `WritingCanvas`
- `ProjectNavigator`
- `AssistantDock`
- `ReferenceDrawer`
- `StatusBar`
- `GhostSuggestionLayer`

### 6.3 状态模型建议

建议至少保留以下 UI 状态：

- `activeProjectId`
- `activeSectionId`
- `activeDraftId`
- `leftNavCollapsed`
- `rightDockMode`
- `zenMode`
- `citationDrawerOpen`
- `unsavedChanges`
- `connectionState`
- `currentLanguage`

### 6.4 数据模型建议

建议的数据关系如下：

- `Project` 包含 `Section[]`
- `Section` 对应当前编辑的 `Draft`
- `Draft` 关联 `Revision[]`
- `Material` 可以被多处引用
- `CitationAnchor` 应保存来源 ID 和范围，不要只保存复制文本

### 6.5 数据接入原则

- 生产环境优先真实数据。
- `simulationData` 只能做开发 fallback。
- 不要在同一界面里混合假数据和真数据而不标记。
- 保存尽量采用乐观更新，但要有失败回退。
- 路由、状态、数据和渲染层不要混成一层实现。

### 6.6 响应式实现原则

- 布局优先使用 CSS 断点和现有栅格。
- 不新增复杂状态库，除非真实复杂度已经证明需要。
- 响应式不是缩小一切，而是重排优先级。
- 窄屏优先保证写作区域可用，辅助模块退后。

### 6.7 性能要求

- 输入响应尽量保持在 100ms 内。
- 面板打开/关闭尽量控制在 200ms 内。
- 路由切换尽量控制在 300ms 内。
- 不要因为动画造成文本输入卡顿。
- 非主路由建议懒加载。

### 6.8 可访问性要求

- 键盘可以走完所有核心路径。
- `Esc` 可关闭抽屉 / 浮层。
- `Tab` 顺序必须合理。
- 图标按钮必须有 `aria-label`。
- 侧边栏收起后仍然能被键盘恢复。
- 必须支持 `prefers-reduced-motion`。
- 状态变化应有可感知的文本或结构提示。

### 6.9 技术边界

- 不引入新的大框架。
- 不用运行时读取 `uiux` 技能文件。
- 不把设计知识做成业务依赖。
- 不新增无必要的状态管理层。
- 不为了“显得高级”而过度抽象。

## 7. 实施阶段

### Phase 1：冻结设计基线

交付物：

- 视觉 token 表
- 布局断点表
- 状态清单
- 路由清单

验收：

- 所有页面风格统一
- 写作宽度明确
- 不再出现风格漂移

### Phase 2：收束布局骨架

交付物：

- `MainLayout` 的三栏 / 抽屉响应式骨架
- 导航与路由对齐
- 侧栏折叠规则

验收：

- 桌面、平板、手机都能用
- 没有死链接
- 写作区始终是中心

### Phase 3：重构写作工作台

交付物：

- `DraftStudio` 拆分
- `WritingCanvas`、`AssistantDock`、`ReferenceDrawer` 等模块化
- Zen mode / citation anchor / status bar

验收：

- 正文不再被辅助模块挤压
- 引用与建议不打断写作
- 状态反馈完整

### Phase 4：真实数据与运行时接入

交付物：

- 资源接口打通
- 运行时事件接入
- 模拟数据降级处理

验收：

- 真数据与假数据边界清晰
- 保存、修订、引用链条可用

### Phase 5：A11y 与体验打磨

交付物：

- 对比度修正
- 键盘导航
- reduced motion
- 空状态 / 错误状态完善

验收：

- 可访问性不掉线
- 细节统一
- 交互克制但完整

## 8. 最终验收标准

这版方案真正通过时，应满足：

- 视觉仍然是现有浅色玻璃风格。
- 写作区宽度稳定在 680–720px。
- 资料、分析、技能入口都还在，但不抢主视觉。
- 左右栏能在窄屏自动退让。
- 引用变得轻量、可追溯、不中断写作。
- 所有状态都有明确表现。
- 文字对比度、焦点态、键盘可用性都达标。
- 没有死路由、没有视觉噪音、没有过度炫技。

## 10. Phase 1 冻结基线确认

本节用于把 Gemini 提出的关键设计决策正式冻结为 Phase 1 的执行边界。

### 10.1 已确认的设计决策

#### 正文宽度与布局

- **确认**：正文区域采用恒定宽度，Phase 1 固定在 `680–720px`。
- **原因**：写作中心优先，保持专注度与阅读节奏。
- **执行规则**：宽屏下保留两侧留白，不在 Phase 1 让正文随窗口无限伸缩。
- **补充**：未来若需要阅读模式或预览模式，可作为独立视图再评估，但不进入本阶段基线。

#### 三栏主次关系

- **确认**：桌面端采用“导航 - 正文 - 助手”的三栏结构。
- **原因**：符合当前工作台骨架，也便于把辅助能力收束到侧栏。
- **执行规则**：资料库、分析、技能等模块在写作中心中只提供摘要、入口或轻量互动。
- **补充**：复杂图表和完整分析能力保留到专门的分析页，不强行塞进窄侧栏。

#### Zen Mode 触发方式

- **确认**：Zen Mode 以**手动开关 + 快捷命令**为主。
- **原因**：这是可预期、可控、可解释的状态切换方式。
- **执行规则**：自动降显可以作为轻反馈存在，但不能作为唯一状态来源，也不能悄悄改变用户界面主状态。
- **补充**：Phase 1 不要求把“持续输入 10 秒自动切换”作为硬规则。

#### 引用标记形式

- **确认**：引用采用正文中的小型锚点 / 角标 / 微型标签。
- **原因**：保持正文清爽，避免引用卡片抢占写作空间。
- **执行规则**：点击锚点后，在右侧辅助区或窄屏抽屉展示来源、原文与关联信息。
- **补充**：引用必须可追溯、可键盘访问、可关闭。

#### 技能注入方式

- **确认**：`uiux` 技能不作为前端运行时依赖。
- **原因**：前端保持轻量，避免把设计知识绑定成业务数据源。
- **执行规则**：技能中的设计建议只转译为静态 token、排版规范、交互原则或文案规范。
- **补充**：后续修改技能配置时，采用人工同步，不要求前端自动热读技能文件。

#### 模块“伪装”清理

- **确认**：不保留“看起来能点、实际上没功能”的入口。
- **原因**：提高真实感，也避免导航信任被削弱。
- **执行规则**：未完成的模块统一标记为 `disabled` / `开发中` / `未启用`，并且禁止假跳转。
- **补充**：对用户可见的入口必须和实际路由或实际能力状态一致。

### 10.2 当前前端代码里仍需冻结的边界

基于 `frontend/` 现有实现，以下边界还需要在进入编码前再次确认：

- `frontend/src/App.tsx` 当前只真正接通了 `/projects` 与 `/draft`；`/knowledge`、`/analytics`、`/skills` 目前是菜单入口而非真实路由。
- `frontend/src/layouts/MainLayout.tsx` 当前存在深色模式切换；需要确认 Phase 1 是否只保留浅色基线，还是把该开关降级为设置项 / 隐藏项。
- `frontend/src/components/DraftStudio.tsx` 仍是三栏原型，但正文区宽度和右侧辅助栏的布局还未完全按本规范冻结。
- `frontend/src/index.css` 里保留了 `.dark` 变量；需要确认这是“未来可选支持”还是“Phase 1 仅保留但不暴露”。
- `frontend/src/types/resources.ts` 目前已有资源类型，但尚未把 `CitationAnchor`、`UIState` 等更细的界面状态模型明确化；这些应在 Phase 2 / Phase 3 再具体落地，除非 Phase 1 需要先冻结接口。

### 10.3 Phase 1 的执行口径

- 先冻结布局与状态边界，再开始细节实现。
- 先保证“写作中心”可用，再追求细分能力增强。
- 先消除死链接和假入口，再优化视觉细节。
- 先统一规范，再拆组件和接数据。

## 11. 定稿结论

**这版前端方案的核心不是换一种风格，而是在你现有风格上，把研究工作台重新排成以写作为中心的秩序。**

如果后续继续推进，建议优先顺序是：

1. 冻结视觉 token 与路由结构
2. 先收束 `MainLayout`
3. 再拆分 `DraftStudio`
4. 最后接真实数据与运行时事件
