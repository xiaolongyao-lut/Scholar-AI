### 处理引擎 (layers/p2_*.py)

#### ✅ p2_claim_extractor.py (Regex + LLM 混合提取)

**职责**: 从文本中提取结构化声明 (Claim)

**核心算法**:
```python
流程: Regex 初筛 → LLM 精化 → 返回声明列表

设计亮点:
1. Regex 初筛速度快 (~100K 文本/分钟)
2. LLM 精化处理边界情况
3. 异步并发处理 (Semaphore 限流)
4. 支持量化属性提取 (value, unit, range)

预期性能:
  - 10K chunks: ~5 分钟
  - 成本: ~$2 (LLM 精化)
```

**关键方法**:
- `extract_from_chunk()` - 从单个 chunk 提取声明
- `_regex_pre_extract()` - 基于模式的初筛
- `_llm_refine_claims()` - LLM 精化（异步）

---

#### ✅ p2_conflict_detector.py (三级语义对齐 + 决策树)

**职责**: 检测两个声明之间的冲突

**核心算法** (【决策 1】实现):
```python
检测流程:
  1. 本地词典查询 (<1ms, 成本 $0)
  2. 向量相似度计算 (<20ms, 成本 $0)
  3. LLM 灰度判定 (仅 [0.65-0.75] 区间, ~1% 场景, 成本 $0.001)

决策树:
  问题: 两个声明的主体相同吗?
    → 否: 无冲突
    → 是: 检查条件兼容性?
          → 不兼容: 条件冲突 (Level 2)
          → compatible: 检查谓词相反性?
                        → 相反: 直接冲突 (Level 3)
                        → 同向: 检查量化冲突?

预期性能:
  - O(n²) 但有分组优化 (按 subject 聚类)
  - 50K claims 对比: ~3 分钟
  - 成本: ~$5-10 (LLM 灰度判定)
```

**关键方法**:
- `align_similarity()` - 三级语义对齐
- `detect_conflict()` - 冲突决策树
- `_load_synonyms()` - 本地词典加载

---

#### ✅ p2_logic_engine.py (全流程协调 + 权威性评估)

**职责**: 协调端到端的逻辑推演流程

**核心能力** (【决策 2】实现):
```python
全流程: [检索] → [声明提取] → [冲突检测] → [权威性评估] → [推演链合成]

权威性评估算法:
  Score = IF × 0.4 + (Citations/10) × 0.3 + Year × 0.3
  
判别逻辑:
  - Score > 0.80: "HIGH" 权威性
  - Score 0.50-0.80: "MEDIUM" 权威性
  - Score < 0.50: "UNCERTAIN" 权威性
  
演变 vs 矛盾判别:
  - 不同年份 + IF/Citation 递增 → EVOLUTION (知识更新)
  - 不同年份 + 不同研究团队 → 需要人工审视
  - 同一年份且矛盾 → CONTRADICTORY (真实矛盾)
```

**关键方法**:
- `reason()` - 完整推演入口
- `_detect_all_conflicts()` - 批量冲突检测
- `_classify_conflict()` - 冲突分类与权威性评估
- `_synthesize_conclusion()` - 最终推论合成

## ⚠️ 已知限制与优化建议

### 限制 1: 正则提取的精度
**描述**: 非结构化长难句的主宾体识别存在偏差  
**影响**: ~15% 的复杂句子可能被遗漏或误提取  
**建议**: 
```
后续优化路径:
  1. 集成专用 NER 模型 (BiLSTM-CRF 或 Transformer-based)
  2. 使用焊接领域的微调模型 (ERNIE + 焊接语料)
  3. 成本: +$2-3/10K chunks
  4. 精度提升: +10-15%
```

### 限制 2: 向量相似度的实现
**描述**: 目前使用字符集重叠系数，未使用真正的语义向量  
**影响**: 复杂表述的对齐可能不够准确  
**建议**:
```
后续优化路径:
  1. 集成 BGE-m3 embedding (已在 P1 中使用)
  2. 本地向量计算无额外成本
  3. 精度提升: +5-8%
```

### 限制 3: 外部 API 容错
**描述**: CrossRef 和 Scrapedia API 未集成 (仅预留接口)  
**影响**: 文献元数据依赖本地数据，可能不够完整  
**建议**:
```
后续优化路径:
  1. 集成 CrossRef API (获取权威作者列表和 DOI)
  2. 集成 Scrapedia (获取实时被引计数)
  3. 成本: +$0.05-0.1/调用
  4. 可用性: 元数据覆盖率从 85% → 98%
```

### 优化机会 1: 并行化
**当前**: 声明提取串行处理   
**建议**: Ray 或 concurrent.futures 并行  
**提升**: 处理速度 3-4 倍  
**成本**: 无额外成本

### 优化机会 2: 缓存策略
**当前**: 每次运行重新计算所有冲突  
**建议**: 实现冲突结果缓存 + 增量更新  
**提升**: 重复查询 10 倍加速  
**成本**: +50MB 存储
