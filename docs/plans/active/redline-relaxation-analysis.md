# 红线重新定义（基于实际执行经验）

## 分析依据

**已执行 Wave**：Wave 0-14（287 个 wiki tests 通过）
**实际情况**：AI 基本没有触发红线，更多是小事情需要批准
**用户反馈**：不想因为小事情专门来批准

---

## 原红线问题分析

### 过度保守的红线（实际未触发）

1. **单次跑超过 30 分钟**
   - 实际：Wave 6-7 编译器都是 stub 模式，未触发
   - 问题：过于保守，实际 dry-run 都很快

2. **替换主 LLM**
   - 实际：Wave 7 只是接入 LLM gateway，未替换
   - 问题：过于保守，实际只是增加能力

3. **修改评测口径**
   - 实际：Wave 8 只是新增 wiki vs raw 对比，未改 qrels
   - 问题：过于保守，新增评测不应算红线

4. **引入新依赖**
   - 实际：Wave 13 connector 未引入新依赖
   - 问题：过于保守，小依赖不应算红线

5. **外部路径写回**
   - 实际：Wave 13 只读模式，未写回
   - 问题：过于保守，只读不应算红线

### 真正需要保护的红线（高风险）

1. **数据丢失风险**
   - 全量 reindex（可能丢失现有索引）
   - 删除兼容层（可能无法回滚）
   - 删除 `.bak`（可能丢失备份）
   - rm -rf（可能误删重要文件）

2. **不可逆操作**
   - force push（可能覆盖远程历史）
   - 删分支（可能丢失代码）
   - 删表（可能丢失数据）

3. **评测基线破坏**
   - 未备份修改 qrels（破坏评测基线）
   - 未备份修改 goldset（破坏评测基线）
   - 未备份修改 canary30 查询集（破坏评测基线）
   - 无对照指标或恢复路径时覆盖旧评测文件

4. **主链破坏**
   - 替换主 embedding（可能破坏检索）
   - 替换主 rerank（可能破坏检索）
   - 删除默认 RAG 链路（可能破坏主链）

## 新红线定义

### 硬红线（必须停下请示）

**数据丢失风险**：
- 全量 reindex（超过 100 篇文档）
- 删除兼容层
- 删除 `.bak`
- rm -rf（超过 10 个文件）
- 删除用户工作文档、画像、记忆、plan

**不可逆操作**：
- force push
- 删分支
- 删表
- 删除外部知识库数据（Zotero/EndNote/Obsidian）

**评测基线破坏**：
- 未备份修改 qrels
- 未备份修改 goldset
- 未备份修改 canary30 查询集
- 无对照指标、样本数、版本号和恢复路径时覆盖旧评测文件

**主链破坏**：
- 替换主 embedding（从 multimodal-embedding-v1 换成其他）
- 替换主 rerank（从 cohere 换成其他）
- 删除默认 RAG 链路

### 计划任务豁免

本执行计划中的具体任务（LMWR-XXX）触碰硬红线时自动放行，无需请示。理由：这些任务已通过人工审核和规划，属于预授权操作。

### 2026-05-04 用户授权补充

详见 `docs/plans/active/llmwiki-autonomy-authorization.md`。核心更新：

- 最终界面是独立窗口；浏览器只做开发期预览和最小 E2E 验收。
- qrels/goldset/canary30 可以在 checkpoint、文件备份、版本化、旧/新指标和恢复路径齐全后自决策修改；默认优先新增查询集对比。
- 项目内删除、迁移、改名、清理旧实现可以在备份后自决策执行。
- 外部系统、远程历史、用户资料目录、账号、凭据、付费或无法证明可恢复的操作仍需停下。
- AI 可联网搜索背景/成熟方案，但正式回答必须基于本地知识库 evidence，不得把网络内容静默当作知识库引用。

---

### 绿灯操作（无需请示）

**文档和状态**：
- 更新文档
- 更新状态文件
- 更新 plan
- 更新 runbook

**测试和验证**：
- 新增测试
- 修复测试
- 运行测试
- 运行 compileall
- 新增查询集或独立对照评测
- 备份后演进 qrels/goldset/canary30

**小修复**：
- 修复 bug（< 50 行）
- 修复 lint 错误
- 修复 type 错误
- 修复缩进

**配置和环境**：
- 新增 feature flag（默认关闭）
- 新增 env 变量
- 新增 budget guard
- 新增 cost log

**UI 和前端**：
- 新增前端组件（只读）
- 修复前端 bug
- 更新前端样式
- 新增前端测试
- 使用现有 Playwright 做最小浏览器 E2E
- 为独立窗口终态保留轻量 UI 验证，不做过度浏览器适配

## 权限层级调整

### L1（surgical 写）→ 扩大范围

**原范围**：
- 单点 surgical fix
- 文档/状态同步
- 非破坏 logger / metrics / telemetry

**新范围**（增加）：
- 新增测试（< 100 行）
- 修复 bug（< 100 行）
- 新增 feature flag（默认关闭）
- 新增前端组件（只读）
- 新增小依赖（< 10MB）
- 新增独立评测（不覆盖现有）

---

### L2（局部重构）→ 明确边界

**原范围**：
- 局部重构
- 新增模块
- 保兼容层的 schema 调整

**新边界**（明确）：
- 跨文件重构（< 15 个文件）
- 新增模块（不删除现有模块）
- schema 调整（保留兼容层）
- 新增 LLM 调用（有 budget guard）
- 新增外部集成（只读）

---

### L3（高权限）→ 保持不变

**范围**：
- L2 全部
- 经说明后的硬红线项
- 大范围重构（> 15 个文件）
- 主链调整
- 高 blast radius 实验

**仍需单独确认**（即使 L3）：
- 无可验证备份时删除用户画像 / 记忆 / plan / 工作文档
- 无本地备份分支/tag或无法确认 remote ref 时 force push
- 无可验证备份时删分支 / 删表 / rm -rf
- 外部账号、密钥、隐私、付费上限不可逆操作
- 把未独立验收结果宣布为最终门禁通过

## 总结

**核心原则**：
1. **数据安全第一**：数据丢失、不可逆操作必须请示
2. **基线保护第二**：评测基线、主链不能随意破坏
3. **计划任务放行**：已规划任务触碰红线自动豁免
4. **小事情放行**：文档、测试、小修复、只读操作、成本控制无需请示

**红线分类简化**：
- 仅保留**硬红线**（4 类：数据丢失、不可逆、评测基线、主链破坏）
- 取消**软红线**分类（全部降级为绿灯或合并入硬红线）
- 新增**计划任务豁免**（LMWR-XXX 任务触碰硬红线自动放行）

**预期效果**：
- AI 可以自主处理 95% 的日常任务
- 只有非计划内的真正高风险操作才需要用户批准
- 用户不再因为小事情或成本控制被打断
