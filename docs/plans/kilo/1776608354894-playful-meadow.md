# 项目执行计划：TOLF 目标架构与当前仓库状态对齐

## 1. 当前判断

- 可以开始执行。
- 但不建议一上来直接切模型或大范围改主线代码。
- 当前最合理的启动方式是：**先修正评测可信度，再验证 reranker 降压，再把成熟组件收敛回主运行时，最后再推进 TOLF 上游能力评估。**

一句话结论：

> **标准 RAG 先作为问题定位和对照组使用；当前优先验证 reranker，而不是先动 LLM。TOLF 仍然是最终目标，但切入点应先放在评测闭环和检索链稳定性。**

## 2. 已确认的项目现状

### 2.1 当前主运行路径

- 主运行时仍以 `pipeline_core.py` + `layers/r_layer_hybrid_retriever.py` 为主。
- 项目级文献接入与 chunk 入库主要落在 `routers/resources_router.py`。
- 前端检索入口主要在 `frontend/src/pages/Workbench.tsx`。

### 2.2 已经存在的较新检索/评测栈

以下能力已经实现，不是空白状态：

- `eval_retrieval_runtime.py`
  - 已支持 BM25/Hybrid + Graph + Dense + RRF + optional rerank + optional expansion。
- `chunk_vector_store.py`
  - 已支持 embedding cache、manifest/hash 硬校验、零向量检测、超长文本切分后均值池化。
- `reranker_client.py`
  - 已支持长文本截断、重试、并发控制、延迟计时。
- `token_utils.py`
  - 已提供 embedding / rerank 的 token 长度保护。
- `smoke_cache_guard.py`
  - 已有 cache/manifest/oversize/zero-row 的 smoke 验证。

结论：

- **长文本安全性并不是完全没做，而是已经做过一轮硬化。**
- 真正需要优先判断的是：这些较新的能力是否应收敛为默认主路径，以及 reranker 是否值得降档。

### 2.3 TOLF 状态

- `layers/tolf_engine.py` 已有较完整实现，不是概念稿。
- `test_tolf_engine.py` 已存在覆盖。
- 但 TOLF 当前更像**可选实验能力**，还不是仓库默认主链路。

结论：

- **TOLF 已具备研究/验证基础，但还不适合在当前阶段直接替换现有主链。**

执行约束：

- **不得回滚现有脏工作区里的其他改动。**
- **后续执行应尽量只触碰与本轮目标直接相关的文件。**

## 3. 已确认的关键证据

### 3.1 旧 baseline 说明 reranker 确实能提效果，但代价高

`BASELINE_METRICS_phase4_with_rerank.json`：

- `Recall@5 = 0.3019`
- `MRR = 0.1762`
- `avg_latency_ms = 3320.37`
- `p95_latency_ms = 5078.88`

说明：

- rerank 对效果有实质收益；
- 但延迟已经明显抬高，符合外部方案文档里“当前主矛盾在 reranker 偏重”的判断。

### 3.2 当前仓库证据并不支持“直接把 8B 换成 4B”

`BASELINE_METRICS_canary30_qwen3.json` vs `BASELINE_METRICS_canary30_qwen3_rerank4b.json`：

- 8B canary：`Recall@5 = 0.1667`，`avg_latency_ms = 20337.0`
- 4B canary：`Recall@5 = 0.1`，`avg_latency_ms = 24040.66`

说明：

- 当前这组仓库内结果里，4B **没有体现出更快且更稳的优势**；
- 因此不能仅依据外部模型选型文档就直接替换 8B；
- 4B 仍然可以作为候选，但必须在**可信评测集**上重跑。

### 3.3 全量 v2.1 评测结果当前不适合作为换模依据

`BASELINE_METRICS_phase5_qwen3.json`：

- `Recall@5 = 0.022`
- `avg_latency_ms = 45022.17`

结合 `docs/superpowers/plans/2026-04-19-gateb-goldset-sampling.md` 已写明的问题：

- 模板 query 高重复度
- 高 fanout
- 旧 evidence_set 更偏单 doc 映射

说明：

- 当前全量评测里有明显 **qrels / 标注结构问题**；
- 这类失真会掩盖真实模型差异；
- 所以**先修评测可信度，再做 reranker 结论**是必要前置条件。

## 4. 执行总顺序

### Phase A：先把评测闭环变可信

目标：先回答“现有指标能不能信”。

优先动作：

1. 以 `docs/superpowers/plans/2026-04-19-gateb-goldset-sampling.md` 为主，建立 Gate B 小金标执行面。
2. 明确使用的小样本 query 集、qrels 格式、pooling 规则。
3. 先用当前 `Qwen3-Reranker-8B` 作为 control 跑小金标或 canary。
4. 在同一 goldset 上再跑 4B / BGE 对照，避免继续用失真的全量结论做判断。

预期涉及文件：

- `docs/superpowers/plans/2026-04-19-gateb-goldset-sampling.md`
- `eval_retrieval_runtime.py`
- `artifacts/eval_audit/*`（如本轮需要产出 goldset / qrels）

通过门槛：

- 能稳定产出同一套 query 的可复现评测结果；
- 结果能够解释 8B / 4B / BGE 差异，而不是继续被 qrels 结构污染。

### Phase B：只动 reranker，不先动 LLM

目标：验证外部方案文档中的核心命题：**当前标准 RAG 下先动 reranker。**

控制变量：

- embedding 暂时固定；
- chunking 暂时固定；
- LLM 暂时不动。

候选组：

1. `Qwen3-Reranker-8B` 作为 control
2. `Qwen3-Reranker-4B` 作为最小改动降档候选
3. `bge-reranker-v2-m3` 作为工程友好型对照组

关键指标：

- `Recall@20`
- `MRR@10`
- `Gold-in-Top-3`
- `Latency p50 / p95`
- `Timeout Rate`
- `413 / zero-vector / cache pollution` 相关异常率

决策规则：

1. 召回不明显下降
2. 排序质量可接受
3. 延迟 p95 明显下降
4. 异常率不上升

若 4B 不满足以上条件，则不切主线。

### Phase C：把成熟的防护和组件收敛回主运行时

目标：减少“评测栈”和“主运行时”分叉。

当前已发现的问题：

- 新的长文本保护、cache manifest 校验、独立 reranker client 已存在；
- 但主运行路径仍有一部分旧逻辑滞留在 `layers/r_layer_hybrid_retriever.py`。

建议执行方向：

1. 将主运行时逐步对齐到 `chunk_vector_store.py` / `reranker_client.py` 的硬化实现。
2. 保持 manifest/hash guard 为默认，不走回退型“静默容错”。
3. 只在 reranker 评测结论稳定后做主链收敛，避免边收敛边换模型导致归因混乱。

预期涉及文件：

- `layers/r_layer_hybrid_retriever.py`
- `chunk_vector_store.py`
- `reranker_client.py`
- `token_utils.py`
- `tests/test_reranker.py`
- `tests/test_dense_rrf_retrieval.py`

### Phase D：补当前 app-path 的近端缺口

目标：保证当前项目的“可运行体验”不被检索研发拖垮。

已观察到的近端风险：

1. `Workbench.tsx` 默认 `ingestMode='query'`，首问可能触发扫描入库，用户体感可能很慢。
2. `settingsStore.ts` 已加入 `retrievalTopK`，但 `Settings.tsx` 看起来还没有完整暴露对应控件，配置链条未完全闭环。
3. `resources_router.py` 对 notebook 失败占位的分类可能不完整，存在把失败内容当可检索文本入库的风险。

建议处理顺序：

1. 先修 notebook 失败判定和扫描链路的边界行为。
2. 再补全 `retrievalTopK` 设置入口与前端配置闭环。
3. 如无强需求，不在这一轮继续扩写前端功能面。

预期涉及文件：

- `routers/resources_router.py`
- `frontend/src/pages/Workbench.tsx`
- `frontend/src/pages/Settings.tsx`
- `frontend/src/services/settingsStore.ts`
- `tests/test_resources_router_scan.py`
- `tests/test_resources_router_contract.py`

### Phase E：再推进 TOLF 上游评估

目标：与外部方案文档保持一致，真正把评估重心前移。

顺序应为：

1. 单元切分评估
2. 图构建评估
3. embedding 在图边/节点召回任务上的专项重评
4. 激活传播与鱼群聚类评估
5. reranker 适配评估
6. LLM 对结构化证据链的消化能力评估

说明：

- 这一步不应早于标准 RAG 基线稳定；
- 否则会同时混入“评测失真 + 主链分叉 + 架构迁移”三种变量。

预期涉及文件：

- `layers/tolf_engine.py`
- `test_tolf_engine.py`
- 未来新增的 TOLF 专项评测脚本与标注集

## 5. 本轮执行的明确边界

本轮不建议一开始就做：

- 直接替换 embedding 主模型
- 直接替换 LLM 主模型
- 在未修评测前宣布 4B 或 BGE 胜出
- 清理或回滚无关脏工作区
- 把 TOLF 一次性并入默认主链

## 6. 推荐的第一批实际落地任务

如果下一步进入执行，建议按下面顺序开工：

1. 固化 Gate B 小金标执行入口，并确认评测输入输出格式。
2. 用现有 8B reranker 跑 control，记录可信基线。
3. 在同一 goldset 上跑 4B 与 BGE reranker 对照。
4. 若出现明确优胜方案，再把主运行时对齐到成熟的 rerank / embedding guard 组件。
5. 然后补 app-path 的 notebook 失败处理与 `retrievalTopK` 设置闭环。

## 7. 最终计划结论

- **现在可以开始执行。**
- **第一阶段不应先做“大改架构”，而应先做“可信评测 + reranker 实证”。**
- **TOLF 是目标架构，但当前工程切入点仍应是标准 RAG 基线下的 reranker 与运行时收敛。**
