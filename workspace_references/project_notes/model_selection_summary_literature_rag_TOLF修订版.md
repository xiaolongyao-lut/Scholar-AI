# 文献分析检索链路模型选型总结（TOLF 目标架构修订版）

## 0. 项目定位声明

本项目的**长期目标架构不是标准 RAG，而是 TOLF**（目标导向、基于信息功能单元、图构建、约束传播与分级鱼群输出的文献提取链路）。

当前文档中对 Embedding、Reranker、LLM 的讨论，部分沿用了标准 RAG 的三层分析方式，其目的**不是**把标准 RAG 作为最终落地形态，而是把它作为：

1. 当前阶段的**参考基线**
2. 低改动成本的**工程对照组**
3. 后续切换到 TOLF 后进行专项评估的**比较起点**

因此，文档中的“标准 RAG 结论”应理解为**阶段性工程建议**，而不是项目最终架构结论。

一句话说清楚：

> **项目目标是 TOLF，标准 RAG 只是过渡期参考系，不是最终落点。**

---

## 1. 这轮讨论的核心结论

这轮讨论的重点，不是“哪个模型榜单第一”，而是：

**文献分析系统应按链路角色来选模型，并且要区分当前参考链路和目标链路。**

### 在标准 RAG 参考链路下

1. **Embedding 层**：负责首轮召回，目标是“别漏掉相关证据”
2. **Reranker 层**：负责候选精排，目标是“把最相关片段排到前面”
3. **LLM 层**：负责基于证据做总结、归纳、结构化写作，目标是“少胡编、分层清楚、输出稳”

### 在 TOLF 目标链路下

系统不再只是“召回 → 精排 → 生成”，而是会演化为：

**单元切分 → 图构建 → 约束传播/上浮 → 鱼群聚类 → 精排 → 结构化生成**

这意味着模型选型不能只看标准三层局部最优，而要看：

- 单元切分质量
- 图边质量
- 传播约束稳定性
- 证据门控能力
- reranker 适配能力
- LLM 对分级证据链的消化能力

---

## 2. 当前项目实际方案与问题定位（基于当前参考链路）

### 当前实际方案

- **Embedding**：`Qwen/Qwen3-Embedding-8B`
- **Reranker**：`Qwen/Qwen3-Reranker-8B`
- **LLM**：ARK endpoint（具体底层模型身份未完全确认）

### 当前方案的优点

1. **体系一致**：Qwen3 Embedding 与 Reranker 同系列，切换、配置、维度和接口逻辑更统一
2. **多语言与长文本能力强**：对中英混排、术语密集、长段落文献更友好
3. **上限高**：适合做高质量离线评测和基线实验

### 当前方案暴露出的主要问题

#### 2.1 Reranker 太重，工程代价高

当前已经明确的工程现象包括：

- reranker timeout 需要从 **15s 提到 45s**
- canary 中 rerank 的 `API p95` 已达到 **15s~18s**
- full eval（3269 条 query）估算达到 **约 88 分钟**
- 单次真实请求仍大致在 **20~30 秒** 量级

这说明当前主矛盾不是“模型不强”，而是：

> **8B reranker 的质量上限虽然高，但在当前工程阶段已经偏重。**

#### 2.2 长文本输入链路曾发生污染

已定位的问题包括：

- 超长 chunk 触发 413
- 旧逻辑会把整批 embedding 回填成零向量
- contextual cache 出现零向量污染
- smoke 最初只能检查 manifest，抓不到零向量污染

这说明文献分析场景下，模型选择不能只看效果，还必须看：

- 长文本安全性
- 超长输入保护
- 缓存一致性
- 脏写风险

#### 2.3 当前问题主要不在 Embedding，而在 Reranker

当前阶段的共识是：

- `Qwen3-Embedding-8B` 不是当前最大问题
- 当前最值得优先调整的，是 `Qwen3-Reranker-8B`

## 3. 模型选型时真正要问的问题

### 3.1 不要先问“哪个模型最强”

应先问：

1. **前 20 个候选里有没有正确片段？**
   - 如果经常没有，问题主要在 **embedding / chunk / hybrid retrieval**
2. **正确片段是否已经进前 20，但前 3 排名经常不对？**
   - 问题主要在 **reranker**
3. **前 5 或前 10 已经有正确证据，但最终回答仍经常答歪？**
   - 问题主要在 **LLM 层**

### 3.2 对文献分析来说，理想链路应是

#### 在标准 RAG 参考链路下

**Embedding 先尽量保召回 → Reranker 再保排序 → LLM 最后保表达**

也就是说：

- Embedding 的要求是“尽量别漏”
- Reranker 的要求是“把对的排上来”
- LLM 的要求是“按证据写，不乱补”

#### 在 TOLF 目标链路下

链路目标会变成：

**切分先保单元质量 → 图构建保边质量 → 传播保激活方向 → 聚类保证据链完整 → Reranker 保代表单元排序 → LLM 保分级证据表达**

也就是说，**标准 RAG 的三层顺序仍然有参考价值，但已经不足以覆盖目标架构。**

## 4. 单独模型推荐（不预设最终架构，只谈能力画像）

下面的推荐仍然保留，因为它们描述的是模型能力画像；但**最终是否采用，必须放到情境化结论里判断。**

### 4.1 Embedding 单独推荐

#### 推荐 A：`voyage-4-large`

**定位：纯 API、效果优先**

适合场景：

- 不想自托管
- 追求高质量多语言检索
- 文献段落较长，希望维度可调、接口成熟

优点：

- 多语言通用检索质量强
- 32K 上下文
- 支持 256 / 512 / 1024 / 2048 维
- 很适合做高质量 API embedding 基线

不利点：

- 纯 API 路线，长期成本较高
- 做 dense+sparse 一体化不如 BGE-M3 直观

#### 推荐 B：`Qwen/Qwen3-Embedding-8B`

**定位：高上限、多语言、长文本友好**

适合场景：

- 中英混排文献
- 长段落 chunk
- 希望和 Qwen3 系列 reranker 或其他 Qwen 生态配套

优点：

- 32K 上下文
- 支持自定义输出维度
- 多语言强，适合学术文本
- 作为强通用 embedding 合理

不利点：

- 更偏通用表达，不像 BGE-M3 那样直接面向检索多功能

#### 推荐 C：`BAAI/bge-m3`

**定位：最贴文献检索栈的 embedding**

适合场景：

- 中文或中英混合文献分析
- 想做 hybrid retrieval
- 希望首轮召回同时兼顾语义相似、关键词命中、术语命中

优点：

- 支持 **dense + sparse + multi-vector**
- 支持 100+ 语言
- 支持长文档到 8192 tokens
- 对“语义 + 术语”并重的任务特别合适

不利点：

- 纯长文本 token 上限不如 Qwen3 32K 宽松
- 如果已经深度绑定 Qwen3 生态，切换成本略高

#### 推荐 D：`text-embedding-3-large`

**定位：工程上最省心的高质量通用 embedding**

适合场景：

- 想快速接入
- 需要稳定 API
- 先求系统跑稳，再做效果优化

优点：

- 多语言能力强
- 维度和成本清晰
- 生态成熟

不利点：

- 更偏通用工业级 embedding，不是检索专项模型

---

### 4.2 Reranker 单独推荐

#### 推荐 A：`rerank-v4.0-pro`（Cohere）

**定位：纯 API、效果优先的 reranker**

适合场景：

- 不想自建 rerank 服务
- 追求高质量精排
- 希望多语言、长片段、半结构化文本都有较好兼容

优点：

- 多语言能力强
- 适合 query + document 联合判别
- 工程接入顺滑

不利点：

- API 成本高
- 大批量离线评测费用明显

#### 推荐 B：`Qwen/Qwen3-Reranker-8B`

**定位：高上限精排**

适合场景：

- 更关注最终精排上限
- 可以接受较高延迟
- 离线评测多于交互式在线查询

优点：

- 与 Qwen3 Embedding 配套
- 多语言、长文本能力强
- 适合高质量基线

不利点：

- 实测已证明：**明显偏重**
- 更适合离线高质量基线，不适合当前在线主力

#### 推荐 C：`Qwen/Qwen3-Reranker-4B`

**定位：Qwen 体系内最值得优先尝试的降档方案**

适合场景：

- 不想完全换体系
- 只想先把最重的 reranker 降下来
- 希望保留 Qwen3 路线的多语言能力和接口一致性

优点：

- 保留 Qwen3 生态一致性
- 大概率比 8B 更适合当前阶段
- 改动最小，验证最快

不利点：

- 仍可能比轻量 cross-encoder 更重
- 需要实际数据验证收益

#### 推荐 D：`BAAI/bge-reranker-v2-m3`

**定位：文献分析里最稳的工程友好型 reranker**

适合场景：

- 中文或中英混合文献
- 想要多语言 cross-encoder
- 更在意部署难度、延迟可控、速度质量平衡

优点：

- 明确面向多语言
- 工程上比大规模 LLM-style rerank 更好控
- 和 `bge-m3` 组合自然

不利点：

- 绝对上限不一定压过所有更大的 reranker
- 更像稳妥主力，不是堆满规格的上限型

#### 推荐 E：`BAAI/bge-reranker-v2-gemma`

**定位：本地/自托管效果优先 reranker**

适合场景：

- 想在 BGE 路线里追更高排序质量
- 接受比 v2-m3 更重的部署代价

优点：

- BGE 官方路线中更偏效果档
- 上限更高

不利点：

- 部署和时延代价更高
- 不是当前阶段最省心的起点

---

### 4.3 LLM 单独推荐

> 注意：生成层是否要动，应在检索层稳定后再判断。

#### 推荐 A：Claude Opus（最新可用高智能档）

**定位：证据约束、复杂分析、长文档归纳**

适合场景：

- 文献综述
- 机理归纳
- 需要区分“原文事实 / 作者解释 / 系统归纳 / 未证实推测”

优点：

- 证据约束型任务稳定
- 结构化输出遵循好
- 更适合高质量分析与写作

不利点：

- 贵
- 慢
- 不适合大批量低成本场景

#### 推荐 B：Claude Sonnet（最新可用高性能档）

**定位：效果与速度平衡**

适合场景：

- 想要 Claude 风格的稳
- 又不想承担 Opus 的代价

优点：

- 结构化输出强
- 比 Opus 更快
- 适合长期工程迭代

不利点：

- 极限复杂分析能力弱于 Opus 档

#### 推荐 C：Qwen2.5-72B / Qwen-Max

**定位：中文学术写作友好**

适合场景：

- 综述写作
- 中文阶段汇报
- 答辩材料
- 学术口吻的中文输出

优点：

- 中文表达自然
- 结构化 prompt 相对稳定
- 适合中文主导的文献分析产出

不利点：

- 证据约束的稳健性需要真实任务验证

#### 推荐 D：DeepSeek-V3

**定位：高性价比批量生成**

适合场景：

- 批量笔记
- 批量文献摘要
- 对成本敏感但仍想要较强生成能力

优点：

- 性价比高
- 适合大批量文献处理

不利点：

- 若任务特别强调“证据约束 + 分层表达”，仍需实测是否达到要求

## 5. 组合推荐（能力画像，不等于最终执行顺序）

### 组合 1：纯效果优先（API 路线）

- **Embedding**：`voyage-4-large`
- **Reranker**：`rerank-v4.0-pro`
- **LLM**：Claude Opus

适合：

- 不计成本
- 追求每一层尽量高质量
- 想快速搭出“效果天花板参考系”

评价：

- 更像采购级最强 API 组合
- 适合做最终高质量参考基线
- 不适合作为当前阶段默认长期主力

### 组合 2：中文学术优先（本地/半本地优先）

- **Embedding**：`BAAI/bge-m3`
- **Reranker**：`BAAI/bge-reranker-v2-m3`
- **LLM**：`Qwen2.5-72B / Qwen-Max`

适合：

- 中文综述、答辩、阶段汇报
- 中英混合文献分析
- 希望检索栈本身更贴文献任务

评价：

- 这是最值得作为长期路线观察的一组
- 原因不是榜单绝对第一，而是**更贴任务形态**
- 如果准备长期做 hybrid retrieval，这套尤其值得认真比较

### 组合 3：保守升级、改动最小（基于当前项目）

- **Embedding**：保留 `Qwen/Qwen3-Embedding-8B`
- **Reranker**：改为 `Qwen/Qwen3-Reranker-4B`
- **LLM**：保持现有 ARK 生成层不动

适合：

- 不想推翻当前代码与缓存策略
- 只想优先解决“reranker 太重”的问题
- 想最快验证“是不是只要降一档 reranker 就够了”

评价：

- 当前最务实、最值得优先试的方案
- 不改变主干架构
- 风险小，定位快

### 组合 4：工程体验优先（全 API、最好调试）

- **Embedding**：`text-embedding-3-large`
- **Reranker**：`rerank-v4.0-fast` 或 `rerank-v4.0-pro`
- **LLM**：Claude Sonnet

适合：

- 早期系统快速迭代
- 想把每层都调得透明
- 先求稳定和可观测性

评价：

- 不是绝对最强
- 但通常是最省心、最好 debug 的一套
- 适合作为对照组

## 6. 建议的适用范围说明

本章建议必须区分两个情境，否则会误把参考链路结论当成目标架构结论。

### 情境 A：标准 RAG 参考链路

用于当前阶段：

- 快速定位问题
- 验证降档方案
- 建立对照组

在该情境下，优先遵循：

> **先动 reranker，不急着动 embedding 和 LLM。**

原因很明确：当前已知主矛盾落在 `Qwen3-Reranker-8B` 的工程代价过高。

### 情境 B：TOLF 目标链路

用于项目长期目标架构。

在该情境下，瓶颈会前移到：

- 单元切分
- 图构建
- 传播约束
- 证据门控

同时：

- Embedding 层会从“纯召回器”扩展为“召回 + 图边权重估计器”
- Reranker 会从“海量 chunk 精排”转向“少量鱼群代表单元精排”
- LLM 需要重点处理“分级证据链输入”，而不是平铺 chunk 列表

因此：

> **标准 RAG 的推荐仅作为迁移期参考，不应直接视为 TOLF 的最终选型结论。**

## 7. 最终建议（按情境重排）

### 7.1 情境 A：维持标准 RAG 参考链路

#### 只做一件事

> **把 `Qwen3-Reranker-8B` 换成 `Qwen3-Reranker-4B`。**

理由不变：

- 当前主矛盾明确在 reranker 延迟
- 改动最小
- 定位最快

#### 若要同时建立对照组

第二优先试：

> **`bge-m3 + bge-reranker-v2-m3`**

理由：

- 更贴文献检索栈
- 适合作为长期路线对照组
- 可以和当前 Qwen 路线做任务级比较

---

### 7.2 情境 B：系统切到 TOLF

在 TOLF 目标链路下，建议顺序要调整。

#### 第一优先：先跑通单元切分和图构建

这是新链路的真正上游。

如果：

- 切分质量差
- 图边噪声大
- 图表绑定不准
- 参数—结果边乱连

那么后面模型再强也补不回来。

所以在动任何模型之前，先把这两件事跑通，并建立评测集。

#### 第二优先：Embedding 层专项评测

TOLF 让 embedding 额外承担图边权重估计职责，不再只是召回器。  
但结论不是“必须换”，而是“必须重评”。

重点比较两件事：

1. **系统是否明显依赖 hybrid retrieval（keyword + semantic 双路）？**
   - 如果是，`bge-m3` 的 dense+sparse+multi-vector 更贴合，应重点对比
2. **系统是否强依赖长文本和 Qwen 生态一致性？**
   - 如果是，`Qwen3-Embedding-8B` 仍有合理性，不必强换

结论应基于真实任务评分，而不是榜单。

#### 第三优先：Reranker 并行降压

TOLF 大概率会压缩 reranker 的有效输入规模。  
但这是“大概率”，不是现在就能写死的结论。

因此工程上最稳的做法是：

> **即使准备切 TOLF，当前阶段仍建议先把 reranker 降到 4B，作为并行减压动作。**

不必等 TOLF 稳定了才动 reranker。

同时注意输入适配：

- 不要把整条鱼群链直接送给 cross-encoder
- 优先采用“**大鱼代表单元 + 少量中鱼上下文**”的输入策略

#### 第四优先：LLM 层暂时不动

TOLF 输出的是分级证据链，LLM 的关键能力会从一般摘要转向：

- 结构遵从
- 低幻觉引用
- 主结论 / 支撑材料分层
- 证据链忠实表达

但这层不应现在就动，建议等检索两层稳定后再单独评测。

如果后续要做 A/B，Claude Sonnet 值得优先进测试池。  
原因不是现在就能断言它一定最优，而是它在“结构遵从 + 证据引用”类任务上更值得优先验证。

---

## 8. 一句话执行结论

### 当前执行结论

> **标准 RAG 下先动 reranker；切到 TOLF 后瓶颈前移，切块和图构建先行，embedding 需专项重评，LLM 最后再说。**

### 更完整一点的说法

> **标准 RAG 是参考基线，不是最终目标。当前优先用它做低成本问题定位与对照组建设；项目最终目标是 TOLF，因此最终模型选型必须以 TOLF 链路下的整体表现为准，而不是以标准 RAG 的局部最优为准。**

## 9. 模型效果评估框架（修订版）

## 9.1 评估框架的前提声明

本项目以 **TOLF 为目标架构**，标准 RAG 仅作为参考基线。

因此，评估框架的作用不是证明“标准 RAG 是否已经足够好”，而是：

- 在当前阶段利用标准 RAG 快速建立稳定对照
- 为 TOLF 的切换提供可比较的基准线
- 在 TOLF 跑通后，将评估重心前移到单元切分、图构建、传播质量与证据链完整性

换句话说：

> **标准 RAG 的评估结果用于“参考和对照”，TOLF 的评估结果才用于“最终决策和长期选型”。**

---

## 9.2 为什么要单独设评估框架

模型推荐只回答“选什么”；  
真正能落地，还必须回答“怎么评、按什么标准决定换不换”。

如果不把评估框架单独补进去，后面很容易变成：

- 凭感觉换模型
- 只看单点榜单
- 把上游错误误判为下游问题

因此，评估必须按链路拆层做，而不是只看最终回答“看起来像不像”。

---

## 9.3 总体原则

统一采用四类指标：

1. **效果指标**：是否真的更准
2. **工程指标**：是否真的更快、更稳
3. **安全指标**：是否减少污染、超长异常、缓存脏写
4. **成本指标**：是否值得长期跑

结论不能由单一指标决定，必须做：

> **效果 — 延迟 — 稳定性** 的三方平衡。

---

## 9.4 情境 A：标准 RAG 的评估框架

### 9.4.1 评估顺序

标准 RAG 下，评估顺序不变：

> **先检索，再精排，最后生成。**

因为当前最大问题明确是 reranker 太重，所以这套评估首先服务于验证：

- `Qwen3-Reranker-4B` 是否能替代 `Qwen3-Reranker-8B`
- `bge-m3 + bge-reranker-v2-m3` 是否值得作为长期路线对照组

### 9.4.2 数据集设计

建议先做一个**小而硬**的评测集，不是一上来全量跑。

推荐：

- 100–200 条真实文献问题
- 每条问题人工标注：
  - 正确证据片段 1–3 条
  - 可接受支撑片段若干
  - 最终理想答案要点
- 问题类型至少覆盖：
  - 参数—结果类
  - 机理解释类
  - 图表证据类
  - 对比归纳类
  - 多文献汇总类

### 9.4.3 Embedding 层指标

Embedding 层只评“有没有把对的捞进来”，不评最终回答。

建议指标：

- **Recall@20 / Recall@50**
- **Hit Rate@20**
- **Evidence Coverage**
- **Chunk Pollution Rate**
- **Embedding Latency p50 / p95**
- **异常率**：413、零向量、缓存污染、重试失败率

判断逻辑：

- 若 Recall@20 明显不足，先看 embedding / chunk / hybrid retrieval
- 若 Recall@20 已够高，不要急着动 embedding

### 9.4.4 Reranker 层指标

Reranker 层评“已经捞到的证据，能不能排上来”。

建议指标：

- **MRR@10**
- **nDCG@10**
- **Top-3 Accuracy**
- **Gold-in-Top-3 Rate**
- **Latency p50 / p95**
- **Timeout Rate**
- **单 query 平均成本**

判断逻辑：

- 如果 gold 已经进前 20，但进不了前 3，就是 reranker 问题
- 如果 4B 能保住主要排序质量，同时显著降低时延，就优先用 4B

### 9.4.5 LLM 层指标

LLM 层只在检索层稳定后再测。

建议指标：

- **Faithfulness**
- **Citation Accuracy**
- **Completeness**
- **Hallucination Rate**
- **Structure Compliance**
- **人工评分**：
  - 可直接用
  - 基本可用但要修
  - 不可用

### 9.4.6 标准 RAG 的决策规则

采用下面这套“先后门”规则：

1. **先过门 1：Recall@20 不降**
2. **再过门 2：MRR@10 / Top-3 Accuracy 可接受**
3. **再过门 3：Latency p95 显著下降**
4. **最后过门 4：异常率不升**

也就是说，4B reranker不是只要更快就行，而是要在：

> **不明显伤召回和排序的前提下更快。**

---

## 9.5 情境 B：TOLF 的评估框架

切到 TOLF 后，评估顺序必须前移。  
因为这时系统不再只是“召回—精排—生成”，而是变成：

> **单元切分 → 图构建 → 激活传播 → 鱼群聚类 → 精排 → 生成**

因此瓶颈前移，评估也必须前移。

### 9.5.1 第一优先：单元切分评估

这是 TOLF 的真正入口。

建议指标：

- **Boundary Accuracy**
- **Function Label Accuracy**
- **Granularity Stability**
- **Figure/Table Binding Accuracy**
- **Redundancy Rate**

建议先做一个小样本人工标注集：

- 20–30 篇文献
- 逐条核对单元边界、标签、绑定关系

### 9.5.2 第二优先：图构建评估

图边错了，后面扩散就会被带歪。

建议指标：

- **Edge Precision@k**
- **Edge Noise Rate**
- **Graph Connectivity**
- **Cross-section Link Accuracy**
- **Link Sparsity / Density**

建议单独抽样看几类边：

- 语义相似边
- 结构邻接边
- 图表绑定边
- 参数—结果边

### 9.5.3 第三优先：Embedding 专项评测

TOLF 下 embedding 不再只是召回器，还承担边权估计职责，所以必须重评。  
但结论不是“必须换”，而是“必须专项测试”。

建议指标：

- **Positive/Negative Edge Separation**
- **Edge AUC / Pairwise Accuracy**
- **Node Retrieval Recall@k**
- **Graph Build Latency**
- **Hybrid Gain**

建议固定对照：

- `Qwen3-Embedding-8B`
- `bge-m3`

判断规则：

- 如果 TOLF 的图边和召回明显受益于 hybrid retrieval，就优先考虑 `bge-m3`
- 如果 dense 语义和长文本能力更重要，且 Qwen 生态一致性有工程价值，就保留 `Qwen3-Embedding-8B`

### 9.5.4 第四优先：激活传播与鱼群聚类评估

这是 TOLF 特有的核心层。

建议指标：

- **Activation Purity**
- **Gold Fish Coverage**
- **Chain Completeness**
- **Chain Redundancy**
- **Activation Drift Rate**
- **Average Fish Count per Query**

这里不要只看单点准确率，要看“链”。

因为 TOLF 的价值不在于某个 chunk 分高，而在于：

> **能不能把参数—现象—结果—图证据这条链带起来。**

### 9.5.5 第五优先：Reranker 适配评估

TOLF 下 reranker 评的是“鱼群代表单元”，不是传统 flat chunk 列表。

建议指标：

- **Big-Fish Top-1 / Top-3 Accuracy**
- **Representative Selection Accuracy**
- **Fish-group nDCG**
- **Latency p50 / p95**
- **输入压缩率**

工程上建议先固定一种输入策略，再评模型，不要混着比：

- 大鱼代表单元
- 附少量中鱼上下文
- 不直接塞整条链

### 9.5.6 第六优先：LLM 对结构化证据链的消化能力

这一层最后测。

建议指标：

- **Hierarchy Fidelity**
- **Evidence-chain Faithfulness**
- **Source Traceability**
- **Over-generalization Rate**
- **Structured Answer Quality**

如果后面要测 Claude Sonnet、Qwen、DeepSeek，这一层再做 A/B。

---

## 9.6 两种情境下共同需要加入的工程指标

无论标准 RAG 还是 TOLF，都建议加三类工程指标。

### 1. 稳定性

- timeout rate
- 失败重试率
- 缓存污染率
- 零向量率
- 图构建失败率

### 2. 时延

- p50 / p95 / p99
- 全链路平均耗时
- 单层耗时拆解

### 3. 成本

- 单 query token 成本
- 单 query API 成本
- 全量评测成本
- 单位效果提升成本

---

## 9.7 建议的最终修订结论

可以把文档最终收束成下面这句：

> **模型选型不能脱离评估框架。标准 RAG 下，评估应按“召回—精排—生成”顺序展开，当前优先验证 reranker 降档是否在不明显损伤排序质量的前提下降低时延。若系统切到 TOLF，评估顺序应前移为“单元切分—图构建—embedding 边权能力—激活传播—鱼群精排—结构化生成”，其中单元切分与图构建是新的上游瓶颈，embedding 层必须专项重评，但不预设必须换型；LLM 层仍应放在检索链稳定后再单独评估。**

---

## 10. 当前可执行动作清单

### 现在立刻做

1. 把 `Qwen3-Reranker-8B` 降到 `Qwen3-Reranker-4B`
2. 建一个 100–200 条 query 的小而硬评测集
3. 固定标准 RAG 参考链路，先跑对照结果
4. 开始搭 TOLF 的单元切分与图构建原型

### TOLF 开发期做

1. 做 20–30 篇文献的单元切分人工标注集
2. 做图边抽样评测
3. 对比 `Qwen3-Embedding-8B` 与 `bge-m3`
4. 设计“鱼群代表单元 + 少量中鱼上下文”的 reranker 输入格式

### 检索链稳定后再做

1. 再评是否需要换 LLM
2. 测 Claude Sonnet / 其他候选模型对分级证据链的结构遵从能力
3. 决定长期主力模型组合

---

## 11. 一句话总总结

> **项目目标是 TOLF，标准 RAG 只是参考基线。当前先用标准 RAG 低成本定位问题、建立对照；最终模型选型和效果评估，必须以 TOLF 链路下的整体表现为准。**
